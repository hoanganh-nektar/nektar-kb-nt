Understand

How Nektar works — from sync behavior and security to key concepts and use cases.

Overview

The Nektar data capture system contains a central database called the Nektar Graph, and three kinds of background services that operate on it: Readers, Builders and Writers.

Nektar Graph

The Nektar Graph is Nektar's representation of reality, containing the activities, opportunities, companies and people from all your tools, as well as the relationships between these objects. Each Nektar customer has a separate, siloed graph.

Readers

Readers are periodic jobs that use APIs to bring in data from external systems like Google Workspace (Gmail, Google Calendar), Microsoft 365 (Mail and Calendar), Zoom, Salesforce, and other integrations. Data that pass read and write rules are added to the graph.

Builders

Builders process the Nektar Graph, performing de-duplication, extracting insights from unstructured text, inferring relationships and applying your business rules. Two key components of builders are graph inference and data transform.

Writers

These are periodic jobs that push changes into Salesforce, updating it to match the reality in your Nektar Graph.

Self-healing

Whenever Nektar gets new data from any of your systems, it re-runs all the above steps, keeping your Graph accurate and comprehensive — even when your graph grows to hundreds of millions of objects. This gives Nektar its self-healing capability.

Graph inference

Nektar's graph inference models identify relationships ("edges") in the Nektar Graph that probably should exist, but don't. This is our core approach: we found a way to reduce all your data problems into missing edges, so edge inference is (very nearly) all we need.

What it does

  • De-duplication of contacts, leads, email messages and calendar events
  • Affinity AI, to identify relationships between data points (such as which opportunity an activity is related to)
  • Edit protection: if a manual Salesforce edit reverts a change that Nektar made, remember the mistake to avoid making it again

How it works

There are many different kinds of edges, so there are many different edge inference models. They all follow a common approach:

  1. First, Nektar identifies candidate object pairs that might have an edge between them, but do not at present. It does this by exploring the neighborhood of each object by "hopping" along existing edges. Depending on the model, it may search up to 3 or 4 hops away from each object.
  2. Nektar then uses a model to get the likelihood of each candidate edge. Inputs to this model include features of the objects themselves (such as timestamps and common text keywords), the number of paths between them, and the number and type of the objects and edges along those paths.
  3. Based on the specific edge type, it creates edges: either for all candidates where the likelihood exceeds a threshold or only for the highest likelihood candidates.

Why it works

Nearly all your data problems can be modeled as missing edges:

  • Duplicate records are just missing "is the same as" relations between the pairs of records.
  • Missing contacts are just missing "belongs to" relations between Salesforce accounts and opportunities and the people your sellers are in touch with.

Self healing

Whenever Nektar gets new data from any connector, it re-evaluates all its past data capture decisions in light of the new information, correcting any errors with the benefit of hindsight. Here are a few example scenarios:

  • The first contact is added to a Salesforce account. Nektar might find old conversations with that contact and add them to Salesforce. Nektar might also find old conversations with others in the same company, and add those contacts and their activities to the account.
  • A lead is added to Salesforce. If Nektar had previously created a contact for this person, its details and activities are merged into the lead and the Nektar-created contact deleted.
  • A contact is added to a Salesforce opportunity. Nektar might reassign the past activities with this contact to this opportunity, and might also add the participants in those activities as opportunity contacts.
  • A preexisting contact is added to an ongoing conversation. Nektar might reassign related emails and events to an opportunity this contact is in.

These are not special cases Nektar is programmed to do, but emergent behaviors of the data processing architecture.

Important notes

  1. For self-healing to work correctly, Nektar might need to undo any of its data capture decisions, including record creation. This is why Nektar requires permission to delete records it created.
  2. Nektar assumes that all manual changes are corrections, never updates. While Nektar's built-in features work correctly under this assumption, it's something to keep in mind when building custom transform rules.

Security

Nektar is a revenue data integration platform. It extracts customer contact and interaction data from the communication tools used by your revenue teams and unifies and links this data to your CRM records.

Infrastructure

Nektar uses AWS infrastructure to host all applications and data storage. Primary servers are in the USA. Amazon provides an extensive list of compliance and regulatory assurances, including SOC 1–3 and ISO 27001.

Data retention and deletion

The default data retention period is 90 days.

  • During normal operations. If a user deletes data from an integrated service (such as Salesforce), the periodic sync job will soft-delete the Nektar copy on its next run. This is the recommended workflow for GDPR deletion requests.
  • On contract termination. An automated offboarding job disables your organization and users and soft-deletes all your data.
  • Soft-deletion marks data as deleted but does not remove it from the database. Soft-deleted data is removed from in-memory caches and is not accessed by the Nektar application.
  • Hard-deletion removes data completely from databases, and happens automatically 60 days after soft-deletion. Database snapshots are automatically deleted 30 days after they are created.

Data usage for AI training

Nektar will not use any customer's data to train its AI and ML models, unless there is a separate written agreement with the customer to use their data for this purpose.

Encryption

  • Data in transit is encrypted using HTTPS (minimum TLS v1.2) with security certificates from trusted providers.
  • Data at rest is stored in AWS RDS with Encryption at Rest using AWS KMS. The database has RBAC via IAM and audit logs of all developer queries.

Compliance

Nektar is certified in compliance with SOC 2 Type II, ISO 27001, and GDPR. Our Terms and Conditions, Privacy Policy, and Data Processing Agreement (DPA) reflect our security readiness.

Sync latency

Nektar's design strikes a balance between efficiency and accuracy, tuned to meet real-world CRM data latency requirements without being wasteful of API quotas or compute resources.

Typical end-to-end latency

Operation Average Worst-case
Creating or updating contacts on Salesforce 1 hour 9 hours
Creating or updating activities with participants already on Salesforce 1 hour 9 hours
Creating activities with contacts that need to be created by Nektar 2 hours 18 hours
Creating Opportunity-Contact relations with contacts already on Salesforce 1 hour 9 hours
Creating Opportunity-Contact relations with contacts that need to be created by Nektar 2 hours 18 hours
Merging Nektar-created contacts with manually created duplicates 25 hours 33 hours

Something appears to have taken longer

This could happen because:

  • Nektar sync might have been disabled or stopped due to error at the time of the event.
  • Nektar sync was only recently enabled for this user; these latencies apply to ongoing operations, not during bulk sync. The sync status icon is blue during bulk sync and green during ongoing operations.
  • Events more than 3 months in the future are only processed when they come within the 3-month window.
  • Auto-recall: if there was no matching account at the time of an email but the account was created later, the account creation is the start-clock event.
  • An event was originally filtered out by a screening rule, but a subsequent edit made it pass. The edit is the appropriate start-clock event.
  • Salesforce API errors may prevent Nektar writing data for prolonged periods, often due to poorly performing Apex triggers or other tools consuming too many API calls.

Sources of latency

There are three main sources of latency:

  1. Nektar fetches emails and calendar events periodically from each inbox and calendar, striking a balance at 10–15 minutes to avoid exhausting Google / Microsoft API quotas. (Average delay: 7 minutes; Maximum delay: 15 minutes)
  2. Nektar aggregates changes over 10–15 minutes to write data to Salesforce in reasonably-sized batches, which is hundreds of times more efficient than writing one at a time. (Average delay: 8 minutes; Maximum delay: 4 hours 15 minutes)
  3. An intentional 30-minute wait before creates and updates ensures that all related records have been independently fetched and unified before writing. Contact merges wait 24 hours.

Data transform

Note

This feature cannot be configured from the console yet. Contact us to set it up.

To write Nektar Graph data to Salesforce, it must be transformed to match your Salesforce data model. This is accomplished using transform rules, which use formulae to calculate the values of Salesforce fields.

When you set Nektar up, a few default transform rules are configured for you. You can edit these, and also add new rules to instruct Nektar to populate additional Salesforce custom or standard fields.

Activity objects

These consist of Events and Tasks. Activity transform formulae can refer to:

  • Fields of the Nektar Graph Activity node (e.g. date and time, subject and body)
  • The list of buyers and sellers associated with the activity, with basic information
  • The record (account, opportunity, etc.) that the activity is associated with

Activity formulae cannot access the history of changes to the associated record, or other activities associated with the same record, buyer or seller.

Note

The information about associated records, buyers and sellers available to activity transform formulae is the latest known information, not the information that existed as of the date of the activity. Transforms are re-evaluated continuously — like Excel formulae. If a field on the associated record is updated on Salesforce, the formula will be re-evaluated and its output written to that Event or Task.

Record objects

These consist of Opportunities, Accounts, Contacts and Leads. Record transform rules can refer to:

  • Fields of the record itself (if it already exists on Salesforce)
  • The list of activities associated with the record, with basic information
  • The list of buyers and sellers associated with the record, with basic information
  • The history of edits made to the record (e.g. stage changes), as tracked by Salesforce

FAQs

Activities

With which account or opportunity would an activity be linked?

Nektar uses proprietary affinity models to determine this. See Graph inference for more information.

An activity is missing in Salesforce. Why?
  1. Activities matching read screening rules are not stored or processed by Nektar, and any that match write screening rules are not written to Salesforce. Verify that no screening rules match the missing activity.
  2. Nektar only captures activities where at least one participant is an existing Salesforce lead or contact, or whose email address has the same domain as an existing contact or account.
  3. Calendar events more than 3 months into the future are not processed by Nektar until they come within the 3-month window.
  4. The activity might be in the processing pipeline. See Sync latency for expected time duration. To look up what happened to a specific activity, use the Tracker.
Do Google Calendar and MS365 Calendar event updates sync to Salesforce?

Yes, Nektar treats Google/MS365 calendars as the main source of truth. Any changes to the meeting title, date, time, participants, or description will sync back to Salesforce. However, if an internal user has made updates to the event in Salesforce, Nektar will not override those changes. See Self healing for more details.

How does Nektar handle activities that occurred before the account was created in Salesforce?

When the account is created and gets a domain name (from the "website" field or a contact), all people with matching email addresses and their conversations will be created on Salesforce. This is part of Nektar's self-healing feature.

Why is Nektar creating old activities now?

Nektar can be configured with a start date, and only activities after that date will be captured. However, Nektar captures activities when it becomes convinced of their relevance to your sales process, which may not be when the activities actually happened. For example, an old activity may be skipped initially and captured later when a Salesforce account is created with a matching domain name.

Does Nektar create duplicate activities in Salesforce?

Nektar does not create duplicate activities. However, Nektar does not de-duplicate against other activity capture systems that may be running on the same users. To ensure no activities are duplicated, disable other activity capture tools for Nektar users.

An activity has both leads and contacts as participants. Who will Nektar create this activity with?

Nektar prioritizes contacts over leads. Nektar will link an activity with a lead only if there are neither existing contacts nor activity participants that can be created as contacts.

An activity has participants from multiple domains. Where will Nektar create this activity?

Nektar creates this activity with the best-suited Account or Opportunity, using proprietary affinity models. See Graph inference for more information.

Why do I see duplicate records when I pull activity reports on Salesforce?

This happens when Shared Activities is turned on in your Salesforce. In shared activities, when internal Salesforce users are added as participants, Salesforce creates a copy of the event for each internal user. To avoid inflated counts, all activity reports should set "Event invitations" to false, ensuring activities are not counted multiple times.

Why is an activity linked to an account even when there are open opportunities under that account?

Nektar's affinity models find the best opportunity or account for each activity. When there are multiple opportunities and there is ambiguity about which one the activity relates to, Nektar adds it to the account instead. If this is happening frequently, review your opportunity and opportunity-contact role creation policies to ensure each conversation has an unambiguous opportunity.

Why are activities assigned to users whose inboxes and calendars Nektar is not reading?

Nektar can only access the inboxes and calendars of Nektar users. However, it may assign emails or events to other Salesforce users if the algorithm determines they are a more appropriate owner. See the Owner documentation for more information.

Contacts

With which account and opportunities would a contact be linked?

Nektar creates contacts under the account with a domain name that matches the contact's email address. An account's domain name may come from its "Website" field, or be inferred from the email addresses of other contacts already in it. Nektar considers a person related to an opportunity if they have participated in an activity related to that opportunity. By default, all buyers related to an opportunity are linked to it using OCRs.

Does Nektar create duplicate contacts on Salesforce?

Nektar will not create a contact when it knows of an existing contact or lead with the same email address. However, it may create a duplicate if it is unaware of the existing record — for example, if the existing contact was created within the last 20 minutes, or if the integration user's permissions prevent Nektar from seeing it. As soon as Nektar becomes aware of a duplicate, it will resolve it by deleting the contact it created and transferring all activities and fields to the surviving record.

I created some contacts on Salesforce. Will Nektar update details like title, phone number, etc.?

Yes, Nektar updates contacts regardless of whether they were created by Nektar or by users.

Admin controls

How can I add or remove users?

See the Users documentation.

How can I stop Nektar capturing private calendar meetings?

You can turn off private calendar meetings sync from the blocking section under read rules.

Can I enable read and write of specific Salesforce objects, e.g. contacts without activities?

Yes, Nektar offers the flexibility to enable read/write at an object level.

Can I see a log of all the activities and persons that Nektar read but did not create in Salesforce?

No. The majority of activity in your sellers' inboxes and calendars is not external sales-related communications. As the volume of such activities tends to be excessive and may contain sensitive or personal conversations, we do not provide an interface to review them manually. If you have a specific activity to track down, use the Tracker.

Security

How does Nektar secure my data?

Nektar is compliant with SOC 2 Type II, ISO 27001, and GDPR. See the Security section for details.

Why does Nektar need delete permissions on Salesforce?

Nektar's self-healing system re-evaluates past data capture decisions in light of new information, and where necessary reverts changes it made. For example, Nektar might add a contact to an opportunity, but later decide a different opportunity was a better fit — in which case it deletes the first opportunity-contact role and creates a new one. At no point does Nektar delete records it did not originally create, or records that were updated manually after creation.

Which IP addresses does Nektar make API calls from?

Nektar will make API calls to your Salesforce, Google, MS365 and other services from one of the following IP addresses: 44.207.41.98, 100.24.77.218, 44.206.28.70, 54.82.117.103.

How many Salesforce APIs does Nektar consume every day?

Nektar is very efficient in its Salesforce API consumption, and on an ongoing basis it consumes under 2,000 API calls (total) per day. For comparison, Salesforce's API quota is 100,000 + (number of licenses × calls per license type) + purchased API Call Add-Ons.

How many objects does Nektar write per API call?

Nektar writes data to Salesforce in batches of 200 objects at a time and continues writing until all objects have been processed. On average, writing 200 objects takes about 10–30 seconds.

How does Nektar's AI agent, Daisy, use and store customer data?

Your data is processed through OpenAI, Google Gemini and AWS Bedrock APIs — all SOC 2 compliant. Enterprise agreements explicitly prohibit data storage or use for model training.
